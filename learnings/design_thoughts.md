EndOfKeyRange property of the storage node directly tells node what range of hashes (hases of key) in the consistent-hash-ring it is responsible for. If a node has EndOfKeyRange as X, then it handles the traffic of hash<=X. FOr more details if there are three nodes in consistent-hash-ring with EndOfKeyRange values of nodea->10, nodeb->50, nodec->100, then nodea handles traffic of all hashes of key <=10, nodeb handles (10,  50], nodec handles (50, 100].

each storage node in the cluster also has gossip behavior, when nodes are added to cluster using /addnode endpoint in cluster, all existing nodes are informed about new-nodes using /updategossip endpoint. when node receives gossip on /updategossip endpoint, they add the new node their their 'gossipNodes' list that they maintain. in the list they maintain information about all known nodes along with the host (ip:port), their id  (uuid), their EndOfKeyRange, and their lastUpdate (timestamp field).

anytime a node recieves get or post traffic, it checks whether it owns the key, that is hash(key) ->token is <= EndOfKeyRange among all nodes, if so then it handles the traffic other wise it find the owner and redirects the traffic to owner nodes and serves the response.

all nodes also gossip on fixed interval letting all known nodes with its own updated timestamp. there fore each node maintains withit all known nodes and their lastKnown update timestamp in cluster (gossip protocol)

In this phase, new node B request source node, say node C, for all data that should be under the ownership of node B. While this is happening, original owner node C will continue to serve reads and write. While the replication is going on, handling read traffic is trivial. Read traffic can be served from original node. However, handling write traffic isn't as trivial. There are few things that comes to my mind when I think of this challenge. First is dual write. In this while the replication is 
going on, all the writes that are coming the owner node will be written to owner node's storage as well as new node's storage. This approach is however prone to a race-condition. Example: assume key K with value V1 is stored in owner node. During replication key K and vlaue V1 is read from owner node and is in flight towards the new node but during that time write comes for K with value V2. This write is successfully written to owner node. So owner node contains K, V2. Also, this write is asynchronously sent to new node. It is possible that new key value (K, V2) beats the original request to replicate key K with value V1 such that the new node has value (K, V2) in store then the original replication request with (K, V1) comes, then the new node will have key value of (K, V1) even though replication has succeeded and original node has (K, V2). To handle this, I am thinking along the lines of versions. timestamps are easy solution to this problem, key and value with higher timestamps wins, however, timestamps do not work in real world distributed systems because of timedrift challenges related to time syncrhonization. Therefore, for versions, I am thinking using counters. The very first time a key and value is written it gets version 1. Subsequent writes on that key increases the version counter (there is still the challenge of atomic writes in storage, but that shouldn't as challenging). Lets go through the original example with this versioning system. Original node and (Key K, Value V1, Version 1) in storage. new node request replication. original node and new node starts the replication process. While the replication is going on, read requests are served by original node. When write request comes for (Key K, Value V2) - original node services the write such that (Key K, Value 2, Version 2) is now stored in original nodes storage. Original node also send aync write request of (Key K, Value V2, Version 2) to new node while the original repication request (Key K, Value V1, Version 1) is still in flight. Similar to previous scenario, the original replication request gets delayed and new request of version 2 reaches the new node first. Therefore the original node has (Key K, Value V2, Version 2) in its storage. And then the original request reaches the new node (Key K, Value V1, Version 1). However when it comes to mutating the storage it will fail because the Version 1 is lower than Version 2. With this approach, the original node and new node is guaranteed to eventually converge to same state (unless the request fails).

Note: i started on point-in-time snapshots before, but then later understood that solving point-in-time snapshots requires persistence to be solved. 1. At present, the point-in-time snapshot of database stops all writes to the datastore, and copies all keys. Goal is to improve this such that, the snapshot doesn't stops any writes.
2. there are several things I am thinking about which might help this. There is a need of WAL, when snapshots are requested, current offsets of wal sets as the boundary for requested snapshot. WAL alone cannot help because in order to create the lastest view of the store upto current_offset from wal will require reading through all mutations in wal which will be time consuming operation. Therefore, there is need of checkpointing, at each checkpoint, the view of store is maintained up until that point. With this strategy, checkpoints + replayed wal will improve the efficiency. This entire thing is actually a solved problem with data-structure called LSM Trees. LSM Trees, however, are little more advanced and slightly different usecase. LSM Trees are designed to store data in disk. They provide fast writes and efficient reads on the disk based storage. My simple approach of checkpoint + wal replay should also work for my usecase as I am providing reads and writes from in memory store. After thinking a lot about it, I have decided to go ahead with checkpoints + wal strategy, and later on, if time permits, I can try LSM Tree for learning. 
3. Detailed Thoughts: For every write request that happens on the store, each mutation/write is appended to wal. Each log in wal contains a offset, key, new value and verion. To get the version of the new mutation, existing value of the key is fetched from the store, if value doesn't exist then version is 1 otherwise, version is existing version+1. After the write to wal is successful, the write is updated to the store. The write is considered successfull, as long as write to wal is success. At this point, I am aware that there is a possibility that the node can potentially fail during the writes to the wal - in which case the wal shall have partial write in it. This situation cannot be prevented, however, the partial writes can be detected using CRC (Cyclinc redundancy checks) which is non-cryptographic hash function to find errors in non-malicious inputs. So, for every write, before data is writte in store, a entry in wal file is made with offset, key, value, crc. After successfull write in wal, value is upserted in store. Write is considered successfull after successful write in wal. After configurable amount of time, checkpoints are created. Checkpoints are full view of the store at the given point-in-time. To create a checkpoint, current_offset in wal is requested, then all the mutations that has happens since previous checkpoint are played on the copy of previous checkpoint to get the new checkpoint. For example, lets say a checkpoint is to be created, to create the checkpoint, current offset of of wal is requested which is, say, 100. If there exist no previous checkpoint, a new in memory store(or map) is created in which on the mutations from beginning to offset 100 are played. The state of the in-memory store is, now, point-in-time view of the store at offset 100. The in-memory store is then converted into file such that keys are sorted. lets call this checkpoint 1. While the checkpoint was being created, store continued to get more mutations/writes which continued without affecting checkpointing or getting affected by checkpointing. When the time again comes to created another checkpoint c2, let say wal is at offset 250. Now, inorder to create the checkpoint, checkpoint c1 is loaded into the memory, and all the mutations/writes that happened after offset 100 and uptill checkpoint 250 are played into the checkpoint c1 which is in memory. After this the in-memory representation of checkpoint c1 contains mutations with offset [0, 100] + (100, 250]. This in-memory representation of c1 can now be saved as file into the disk with as checkpoint c2. This process will be repeated indefinitely, until needed. Old checkpoints can be removed if not needed. There are verious ways to remove old checkpoints, such as limiting the number of past checkpoints or limiting the max age of the checkpoints. There is a serious flaw in this strategy however, creating new checkpoints requires loading into memory old checkpoint, which for small data footprint is achieveable, however, as the size of the data grows will require more and more ram (which is super expensive now a days). 
4. After doing some search, I think, there is different approach to create checkpoints. In this strategy, when a checkpoint is requested, current_offset is requested from wal. Then, a new sstable is created from mutations ranging from (prev_checkpoint_offset, current_offset]. Here, prev_checkpoint_offset is the offset will which previous checkpointing has been done. If no checkpoints has been created before then prev_checkpoint_offset is the beginning of the wal (likely 0). the SSTable created from mutations in range (prev_checkpoint_offset, current_offset] is sorted on keys. Previous checkpoint, if any, is also a SSTable sorted on keys. Then, the problem of creating new checkpoint boils down to merging two SSTables which is a solved problem. Merging to SSTables is similar problem statement as that of merging to sorted arrays. This can be achieved on disk directly without having to load any significat portion of files into the memory. After the operation of creating new checkpoint is completed, the new checkpoint is available for use. Present strategy allows the creation of checkpoints efficiently for arbitrarily large checkpoints. However, another challenge with present strategy is every growing wal size. When the threshold is breached, then mutations/writes are written to new wal file. The offsets in each file begins from 0 (at this point I see no reason to have always increasing offsets) and each new wal file can be named with <timestamp>_wal_v1.txt. 2. For replica's - there are following things that needs to be defined. ReplicaCount, ReadQuorum, WriteQuorum. ReplicaCount is number of addition node which have copy of the node. If ReplicaCount is 2, that means in addition to owner-node there are 2 additional nodes in the cluster which have the copy of the data. WriteQuorum is how many nodes including owner-node must confirm write for write to be successfull, WriteQuorum has valid values from 1 to <= ReplicaNode+1. For WriteQuorum=1, writes are confirmed for owner-node, for WriteQuorum=2 writes are confirmed for owner-node and 1 additinoal replica. In this approach, it might be possible that WriteQuorum was not met as the requires number of replicas were not available or failed to return response in time - in those scenarios writes are persisted in owner-node (with the goal to replicate later on, however, I've not yet given mush thought on how to sync the replicas with owner-node). As long as owner-node commits write in it, it will be considered success. ReadQuorum can have values in [-1, ReplicaCount]. The value of -1 represent read from owner-node is requested. In this case, the most recent view of the data is guaranteed. For ReadQuorum=1 means read from any one node among owner and replica, this is default behavior, ReadQuorum=2 means read from any two node among owner and replica, in case there are two different versions are returned by different node, most recent version is returned. ReplicaCount is cluster level construct, this means all nodes in the cluster will have same ReplicaCount. ReadQuorum and WriteQuorum are request level construct.

READ_QUORUM has valid values from [-1, REPLICA_COUNT]. For READ_QUORUM of -1, the value of requested key is fetched from owner-node, for READ_QUORUM of 0 value is fetched from the current node whether or not current node is owner-node or replica-node or not. For example of a request to get key=K lands of a arbitary node N then the key is looked up in node N's store. For READ_QUORUM value of R where R belong in range [1, REPLICA_COUNT] the R random replica's are requested for values, the value with most recent version is returned. WRITE_QUORUM has valid values in range [-1, REPLICA_COUNT]. Irrespective to the value of WRITE_QUORUM the value is first written to the OWNER_NODE. Then the key, value, version are written to REPLICA_NODES. If the write to owner-node fails for any reason, complete write is considered as failed. This is a flaw that is present in the present architecture, in future, improvements can be made such that if owner-node is unavailable, the next available node accepts the write, and as and when new owner-node becomes available (or new node is chosen as owner) the write is propagated to that node. For the WRITE_QUORUM value W of -1, the value is awated replicated to all replica nodes and awaited for confirmation by all until timeout occurs which is defined by WRITE_CONFIRMATION_TIMEOUT. It may be possible that not all nodes return ack within expected time frame in which case nothing is can be done but the response is returned to user about how many replicas were able to confirm the write. As of this moment there exist no mechanisms which can ensure that out of sync replicas can sync it self (Maybe a exercise for future). For WRITE_QUORUM W such that is belong in range [0, REPLICA_COUNT], W number of replicas are awated for write confirmation until timeout occurs.
